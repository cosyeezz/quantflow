# Product Guidelines

## 1. 设计与用户体验 (Design & UX)
- **核心策略**: **前端驱动的复制与验证 (Frontend-Driven Copy & Verify)**。我们从现有的 `easyquant` 项目迁移设计，但这不仅仅是复制粘贴。
- **视觉风格**: **已有设计迁移**。基于原有设计体系，但在迁移过程中进行必要的现代化调整。
- **交互原则**: **体验升级 (UX Upgrade)**。我们不应盲目复刻旧的交互模式。在保留核心功能的前提下，积极寻找机会优化用户流程（例如减少点击、优化拖拽反馈），致力于提供比原版更流畅的操作体验。

## 2. 工程标准 (Engineering Standards)
- **代码质量**: **严格规范 (Strict Standards)**。此次重构是消除技术债的关键机会。所有新迁移或编写的代码必须严格遵守 Linting 规则和类型定义（Type Safety）。不允许引入 `any` 类型或忽略 Lint 警告，除非有极度充分的理由。
- **重构原则**: **精简 (Lean)**。严格审查旧代码，剔除死代码、冗余依赖和过时的业务逻辑。只迁移当前 UI 真正需要的功能。

## 3. 开发流程与协作 (Workflow & Collaboration)
- **API 设计**: **API First**。前端需求驱动后端开发。
    1. 前端分析 UI 需求。
    2. 定义清晰的 API Schema (OpenAPI/Swagger)。
    3. 后端依据 Schema 实现接口。
- **验证机制**: 每个迁移的组件必须在前端可交互验证后，才视为完成。
